**Operational pattern (owner's observation):** shanti's workflow appears to be copy-forward per kelas each year — a student promoted from KB1 to A1 the next year gets their row copied into the new kelas sheet, keeping the same NIS (assigned once at entry, per shanti's own explanation of the NIS-year-prefix rule). It also means **this file will recur** — next year's roster file will re-send most of these same 140 students (promoted a kelas, same NIS). Any import must be idempotent keyed on NIS: re-running against a future file should update/skip already-imported students, not duplicate them. This is a hard requirement for T2, not just a nice-to-have.

## Implementation

- Subagent plan: driver=claude-sonnet-5, dirty-work=claude-sonnet-5 (single general-purpose subagent — cycle too small to split T2 further; driver reviewed + sent back one fix-round rather than doing a parallel fan-out).
- Task T2: `scripts/roster-import/{parse-xlsx,map-fields,dedupe,config,run}.ts` + `scripts/roster-import/__tests__/*.test.ts` — Prisma-direct idempotent import script, dry-run by default. Architecture note: deviates from the prior cycle's "drive the admin API from the browser" approach — this repo's own `scripts/reseed-staging.ts` / `scripts/reseed/people.ts` precedent uses Prisma directly for bulk data ops, and `reconcileSessions` (the side effect the API route triggers on `ClassSection` create) is a plain importable function from `lib/sessions/reconcile.ts`, not something locked behind the HTTP layer. Chosen because this import recurs yearly (140 vs the prior one-off's 35) and a committed, testable script is worth it here.
  - `config.ts` hardcodes the 11-of-13 resolved campus mapping; `DC` and `KB2` deliberately absent — the script warns and skips any kelas not in the map rather than guessing.
  - `dedupe.ts`: parent reuse requires the record's ayah+ibu names to match an existing family's ayah+ibu pair *together* (not either name independently) — a code-review pass caught the independent-match version as a blocker (risk: two unrelated families sharing one common first name get cross-wired).
  - `run.ts --commit` requires `--academic-year=` explicitly and throws rather than creating/guessing an `AcademicYear` — T1's AY question is still open.
  - Each student's writes (Parent create-or-reuse, Student, StudentGuardian, StudentEnrollment) run inside one `prisma.$transaction` so a mid-import failure can't leave an orphaned half-created Student that idempotency would then permanently skip.
  - Error logs redact PII (NIS/kelas/row index only, never raw NIK/address/income) per code review.
  - Code review (`superpowers:code-reviewer`) found 3 blockers pre-fix (parent dedup independent-name matching, missing primary-guardian fallback for mother-only rows, no per-student transaction) — all fixed and re-verified independently (build + scoped vitest run) before commit, not just trusted from the subagent's self-report.
  - Bonus catch while writing `parse-xlsx.ts` tests: header-row detection was anchoring on the literal text "NIS", which appears in both the super-header and sub-header rows of the real sheets — was silently picking the wrong row. Fixed by anchoring on "Lengkap" instead (unique to the true sub-header). Re-verified against the real source file post-fix.
- Post-review addition, driver-authored (not delegated — small and precise enough that writing the brief would have cost as much as writing the code, given the real-prod stakes): `scripts/roster-import/overrides.ts` + `scripts/roster-import/__tests__/overrides.test.ts` + edits to `run.ts`.
  - `isExcluded`/`isWithdrawn`/`noGuardianOk` match on normalized `kelas::name`, resolved directly against shanti's WhatsApp messages (see Spec's blank-NIS bullet).
  - `TD1_MANUAL_RECORD` injects the one TD1 student (no `Data TD1` sheet exists to parse) as a synthetic `RosterRecord`, and is the sole record exempted from the "must have ≥1 guardian" invariant.
  - `run.ts` asserts exactly 2 excluded + exactly 2 withdrawn records post-filter and throws loudly if either count is off — a code-review finding flagged that a typo in a hardcoded override name would otherwise silently no-op (excluded student imported normally, or withdrawn student imported as ACTIVE) with no error signal on a real prod PII write.
  - Code review (`superpowers:code-reviewer`) on this addition found 3 issues, all fixed: (1) TD1's name/NIS weren't traceable from the cycle doc alone — added an explicit "Identity, for traceability" note to the TD1 acceptance-criterion bullet in Spec; (2) doc-sync gap — this Implementation/Verification update; (3) no collision guard against a hardcoded-name typo — the exact-count assertions above.

- Task T3, execution mechanism (deviation from T2's plan): no `DATABASE_URL`/prod credentials are available in this harness's environment, so `run.ts --commit` (Prisma-direct) could not be executed as originally planned. Delivered via a hybrid instead: (1) the 13 `ClassSection` rows were created by driving the real prod admin UI (`https://talib.annisaasekolahku.com/admin/classes`) via browser automation, one at a time, confirmed via dropdown selections against the app's own Campus/Program values — this lets `reconcileSessions`' calendar logic run server-side exactly as the app would trigger it, rather than hand-replicating that logic in raw SQL; (2) Parent/Student/StudentGuardian/StudentEnrollment rows were generated by a one-off script (`scripts/roster-import/generate-sql.ts`, not part of the reusable suite) that reuses `parse-xlsx`/`map-fields`/`dedupe`/`overrides`/`config`, emits raw SQL, and was executed directly against prod (`vxwywmvpxetdgnxejjgk`) via the Supabase MCP `execute_sql` tool in small chunks (tool-call size constraints — ~40-83 statements per chunk, sequenced Parents → Students → Guardians → Enrollments to satisfy FK order; no single wrapping transaction across chunks, so idempotency/rollback relies on the NIS-keyed dedup snapshot and a verified pre-check that nothing partial existed before starting).
  - `generate-sql.ts` corrected 3 hardcoded values that turned out wrong when checked against live prod data rather than trusted from source files: `TENANT_ID` is `"tenant_annisaa"` (not `"t_annisaa"` from `scripts/reseed/org.ts`), Campus names are `"Taman Aster"` / `"Metland Cibitung"` (not `"An Nisaa' Sekolahku Taman Aster"` / separate `"D'Care (Day Care)"`), and there is a single `"TK Islam Terpadu"` Program with `ageGroup` distinguishing A/B (not separate `"TKIT-A"`/`"TKIT-B"` programs). All three were caught by cross-checking the app's own admin-UI dropdowns and direct SQL before any write, not assumed.
  - **Data-corruption bug found and fixed mid-execution:** `parse-xlsx.ts`'s header-mapping had a latent bug — the `Data KB4` sheet's row-8 "Tempat"/"Tanggal" sub-header cells are merged (`H8:I8` in the real file), and ExcelJS inherits the merged-cell's value onto every cell in the merge range (unlike openpyxl, which returns `None` for non-master cells). This silently overwrote `colMap.birthPlace` with the `birthDate` column's index for that sheet, so every KB4 student's `birthPlace` field held a raw ISO timestamp string instead of a city name, and `birthDate` was `null`. Caught by manual inspection of the generated SQL output (an ISO-looking string in a `birthPlace` column is an obvious tell) before most of the affected batch had executed. Root-caused by inspecting the real workbook's merge ranges directly, reproduced with a standalone script against the live file, fixed with a guard-first-match + positional fallback in `parse-xlsx.ts`, and covered by a new regression test (`__tests__/parse-xlsx.test.ts`) that reproduces the exact merge pattern. The ~62 students + ~230 parents already written under the buggy parser were verified to have zero dependent rows (no guardians/enrollments referencing them yet) via SQL, then deleted, and the corrected script was regenerated and re-executed from scratch.
  - Prod's exact `ClassSection.id` values (13, one per kelas) and `Campus`/`Program` ids were captured as hardcoded constants in `generate-sql.ts` after the UI-driven creation step, since this script is a one-off for this cycle's data, not a reusable idempotent tool (T2's `run.ts` remains the reusable path for a future recurrence of this same file, once a `DATABASE_URL` is available in whichever harness runs it next).

## Ship Notes

- **No schema migration** — this cycle is a data-only import (raw SQL INSERTs against existing tables) plus 13 new `ClassSection` rows created through the admin UI. No Prisma migration file, no `prisma/schema.prisma` change.
- **No new env vars.**

  - Soft-rollback (preferred, matches the prior cycle's precedent): set the affected `Student.status` to `WITHDRAWN` via the admin UI rather than deleting rows, to preserve audit history and avoid FK cascade surprises.
  - Hard-rollback (only if truly necessary, e.g. a bulk mistake caught immediately): all rows from this cycle are identifiable by the `cimp` id prefix used throughout (`Parent`, `Student`, `StudentGuardian`, `StudentEnrollment` — see the verification queries above for exact `WHERE id LIKE 'cimp%'` scoping). Delete in FK-safe order: `StudentEnrollment` → `StudentGuardian` → `Student` → `Parent` (only `Parent` rows not shared with the pre-existing 34-student cohort or the 3 sibling families — check `StudentGuardian` references before deleting any `Parent`). The 13 `ClassSection` rows created via the admin UI are not `cimp`-prefixed (real cuid()s from the app) — their ids are hardcoded in `scripts/roster-import/generate-sql.ts`'s `CLASS_SECTION_ID_BY_KELAS` map if a full rollback of those is ever needed; deleting a `ClassSection` also removes its generated `ClassSession` rows.
