**AC-3 — `unstable_cache` on parent helpers + admin dashboard count**

`lib/parent-helpers.ts` — both functions wrapped with `unstable_cache`:
```ts
export const getParentWithChildren = unstable_cache(
  async (session: SessionUser) => { /* existing Prisma query */ },
  ["parent-children"],
  { revalidate: 300, tags: ["parent-children"] }  // 5 min TTL
);

export const getStudentInvoices = unstable_cache(
  async (studentId: string) => { /* existing query */ },
  ["student-invoices"],
  { revalidate: 120, tags: ["student-invoices"] }  // 2 min TTL
);
```

`app/admin/page.tsx` — `totalEmployees` count wrapped:
```ts
const getEmployeeCount = unstable_cache(
  async (tenantId: string) =>
    prisma.employee.count({ where: { tenantId, status: "ACTIVE" } }),
  ["employees-count"],
  { revalidate: 1800, tags: ["employees-count"] }  // 30 min TTL

```

Cache invalidation wired via `revalidateTag`:
- `app/api/xendit/webhook/route.ts` → on PAID: `revalidateTag("student-invoices")`

- `app/api/invoices/[id]/route.ts` PUT → `revalidateTag("student-invoices")`
- `app/api/employees/route.ts` POST → `revalidateTag("employees-count")`
- `app/api/employees/[id]/route.ts` PUT (status change) → `revalidateTag("employees-count")`

### Assumptions

1. **`unstable_cache` key uniqueness:** `getStudentInvoices` uses `["student-invoices"]` as the key
   array but different `studentId` values produce different cache entries because Next.js includes
   the function arguments in the cache key automatically.
2. **`attendance/monthly` revalidate:** This route accepts query params (`?month=&year=`). Route-level
   `revalidate` caches per-URL including query string — no stale cross-month data risk.
3. **Phase 3 indexes on prod:** I'll apply them via a single combined SQL statement in
   `apply_migration` rather than re-running the old migration file, to avoid `_prisma_migrations`
   table conflicts on prod.
4. **RLS fix scope:** Only policies calling bare `auth.uid()`. The `_service_all` policies use
   `true` — no change needed there.
