# Admin Raport MVP — Auto-Drafted Triwulan Report Card

## Context

The `/admin/penilaian` surface (cycle 2026-06-01) is a deliberately read-only completion *monitor* over `AssessmentEntry` — it shows how many students each walas has assessed, but exposes no per-student data and no report-card management. To an admin it reads as half-built: "where do I actually see and manage the raport?" The master design (`docs/archive/superpowers-legacy/specs/2026-05-12-curriculum-penilaian-raport-design.md`) specifies a full **Raport Triwulan** subsystem (C8–C11): per-student narrative report card aggregating penilaian into 5 narrative sections + closing, kisi-kisi template authoring, teacher/walas entry, admin publish, PDF/docx, parent download + e-sign. That is ~6 phases and cannot land in one cycle. This cycle ships the **thin admin-first vertical slice**: a new `/admin/raport` where an admin picks a triwulan + class, sees each student's report card **auto-drafted from existing penilaian** (dominant `AchievementLevel` per curriculum element + auto-pulled attendance), **overrides any field**, publishes, and downloads a PDF. It makes the penilaian→raport loop real for the admin now and lays the schema (`Term`, `ReportCardEntry`) the later phases build on. Decisions below were locked in a brainstorming pass with the user.

## Spec

**Acceptance criteria**
- [ ] New admin surface `/admin/raport`: Term + Class selector → roster table with per-student raport status (Belum dibuat / Draft / Terbit) and an edit action. Gated by `reportCard.read` (SUPER_ADMIN + SCHOOL_ADMIN).
- [ ] Opening a student with no saved entry shows an **auto-drafted** report: each narrative section's `AchievementLevel` pre-filled with the *dominant* level from that student's `AssessmentEntry` rows in the Term window (`voidedAt IS NULL`), and attendance counts auto-pulled from `StudentAttendance` over the same window. Draft is not persisted until saved.
- [ ] Section→penilaian mapping: RELIGIOUS_MORAL ← RELIGIOUS_MORAL, IDENTITY ← IDENTITY, STEAM ← STEAM, PERFORMANCE_SHOWCASE ← MOTOR_SKILLS + ART pooled, INTRODUCTION ← no suggestion (manual). Each suggested level shows the raw rollup counts beside it ("saran penilaian") so the admin sees why.
- [ ] Admin can override every field — section levels, per-section narrative text, attendance counts, hafalan notes, height/weight — and save (`PUT`). Save upserts `ReportCardEntry` (+ `StudentMeasurement`).
- [ ] Admin can Publish / Unpublish an entry (`status` DRAFT↔PUBLISHED + `publishedAt`). Gated by `reportCard.publish`.
- [ ] Admin can download a per-student PDF report card via `@react-pdf/renderer`: school+student header, 5 narrative sections (label + level + text), attendance table, hafalan, measurements, signature lines.
- [ ] New "Raport" item under the admin "Penilaian" nav group; `/admin/penilaian` keeps its monitor role and cross-links to `/admin/raport`.
- [ ] All raport API queries tenant-scoped per `security.md`; mutations zod-validated per `api.md`.
- [ ] Between-task gate green every task: `npm run build && npx vitest run`. End-of-cycle: + `npx playwright test`.
- [ ] Frontend diffs cross-checked against `design-system.html` (admin list/detail, form fields, status colors); recorded in Verification.

**Non-goals** (explicit later-cycle phases)
- Kisi-kisi narrative templates (`ReportNarrativeTemplate` / `ReportClosingTemplate`) — narratives are admin-typed free-text this cycle.
- Teacher/walas authoring workflow (`/teacher/raport`) — entry is admin-driven this cycle.
- Parent download / comment / e-sign — columns added to schema, no UI/route wired.
- docx output (`docxtemplater`) — PDF only.
- Hafalan structured tracking — free-text `memorizationNotes` only.
- `homeroomTeacherId` / `isHomeroom` on `ClassSection` — `ReportCardEntry.homeroomTeacherId` is nullable, admin-set.
- Dropping/altering legacy assessment tables; sentra rotation; any change to existing `/admin/penilaian` monitor logic.

**Assumptions** (correct me before /build)
1. **Term setup is minimal.** Admin creates a Term (active semester + number 1|2 + start/end dates) via a lightweight form on `/admin/raport`; no separate `/admin/terms` page this cycle. Term dates define the aggregation + attendance window.
2. **"Dominant level" = mode** of the student's entries for that element(s) in the window; tie broken toward the lower achievement (NEEDS_REINFORCEMENT > EMERGING > CONSISTENT precedence) to avoid over-stating. No entries → section starts blank (no suggestion).
3. **Attendance mapping:** for the student's `StudentAttendance` rows in the Term window, `status` PRESENT|ABSENT|SICK|PERMISSION → sickDays=count(SICK), permittedAbsenceDays=count(PERMISSION), unexcusedAbsenceDays=count(ABSENT). `totalSchoolDays` = count of distinct dates in the window that have ≥1 `StudentAttendance` row tenant-wide (proxy for school operating days, so it isn't undercounted by a student's missing rows). Admin overrides any.
4. **Permissions:** new `reportCard.read` / `reportCard.write` / `reportCard.publish`, all granted to SUPER_ADMIN (owner escape hatch) + SCHOOL_ADMIN. Matches master design §3.2 publish row.
5. **`sectionLevels` + `sectionNarratives` are JSON** on `ReportCardEntry` (5 fixed sections; atomic per-entry write) per master design rationale. Parent-sign + walas/kepala-sign columns added now (nullable) but unused.
6. **Schema delivery:** additive Prisma migration; no destructive change. Pilot prod DB is near-empty (per pilot-rollout memory), so migration risk is low.
7. **Commit hygiene (commit-msg hook):** every `feat:` task touching `app/**` or `lib/**` stages `README.md` in the same commit (folds its module/route/entity line inline). T6 carries CLAUDE.md counts + `/audit-docs`.

## Ship Notes

**Migrations:** ONE new additive migration `prisma/migrations/20260606000000_add_raport_models/` — creates `ReportSection` enum + `Term`, `ReportCardEntry`, `StudentMeasurement` tables (+ indexes, FKs, service_role RLS). No destructive ALTER/DROP, no backfill. CI/Vercel runs `prisma migrate deploy` on a fresh DB; on prod/staging it applies cleanly (pilot DB near-empty). Rollback = revert commits + drop the 3 tables + enum (no data to preserve).

**Env vars:** none new. PDF uses the existing `@react-pdf/renderer` dep + `NEXT_PUBLIC_APP_URL` (already set).

**Permissions:** new `reportCard.read`/`reportCard.write`/`reportCard.publish` granted to SUPER_ADMIN (escape hatch) + SCHOOL_ADMIN. **System-role seeding:** if existing custom/system roles are persisted in DB with a frozen permission array, they will NOT auto-gain `reportCard.*` — SUPER_ADMIN works via the escape hatch regardless; SCHOOL_ADMIN via its code-derived defaults. Re-seed/refresh role permission arrays if a tenant relies on persisted custom roles.

**Behavior changes users will notice:**
- Admin sidebar "Penilaian" group gains a **"Raport"** item → `/admin/raport` (triwulan report card: pick triwulan + class → per-student raport auto-drafted from penilaian, override any field, Simpan / Simpan & Terbitkan / Unduh PDF). The penilaian monitor header gains a "Susun Raport" cross-link.
- First use requires creating a **Triwulan (Term)** on `/admin/raport` (semester + number 1|2 + start/end dates) — defines the penilaian + attendance aggregation window.

**Preview-verify focus (for `/ship`, signed-in Google session):**
