# Performance Optimization — Phase 4: Production DB Sync, FK Indexes, RLS Initplan & unstable_cache

## Context

Phases 1–3 delivered: N+1 query fixes, dashboard parallelization, 9 reference routes cached,
4 Prisma indexes (User/PayrollItem/Admission/ClassSection), payroll queries optimized,
attendance/today 2→1 round trip, payroll writes createMany bulk transaction.

This cycle was triggered by a fresh Supabase Advisor audit (2026-04-15) and a cross-DB index
comparison between production (Mumbai `qrnbanxcrmrwganpmzmn`) and staging (Tokyo `jzhujpqaxyeeokgexerc`).

**4 problems found, ordered by impact:**

**1. Production DB is behind staging (critical):**
The Phase 3 migration `20260415000000_add_perf_indexes` applied to staging but never to production.
Production is still doing full-table seq scans on User, PayrollItem, Admission, ClassSection —
the exact queries the migration was designed to fix.

**2. 8 unindexed FK columns on production (Supabase Advisor, INFO-level):**
Supabase flagged 10 FK columns without covering indexes. 2 of those (`User.tenantId`,
`PayrollItem.payrollRunId`) will be resolved when Task 1 lands. The remaining 8 are new:
- `Campus.tenantId_fkey` — campus list + RLS filter
- `Employee.campusId_fkey` — employee list filtered by campus
- `EmployeeSalaryValue.componentDefId_fkey` — payroll generate includes all salary values
- `LeaveRequest.employeeId_fkey` — leave page JOINs on employee
- `PayrollItem.employeeId_fkey` — Phase 3 added `(payrollRunId, employeeId)` composite; that covers `payrollRunId_fkey` but NOT `employeeId` alone
- `PayrollItemLine.componentDefId_fkey` — payroll detail page fetches all lines per item
- `PayrollItemLine.payrollItemId_fkey` — same detail page
- `PayrollRun.tenantId_fkey` — payroll list filters by tenant

**3. 15 RLS initplan warnings on production (Supabase Advisor, WARN-level):**
Policies on 13 tables call `auth.uid()` or `current_setting()` per-row instead of once per query.
Fix: wrap in `(select auth.uid())`. Note: Prisma uses the service role and bypasses RLS, so this
does not affect current Prisma query performance. The fix is best-practice hygiene for when we add
Supabase Realtime or direct-client access. Low implementation risk; pure SQL.

**4. Server components re-query DB on every request (medium, no caching):**
`app/parent/page.tsx` calls `getParentWithChildren(session)` and `getStudentInvoices(studentId)`
on every load. These are Prisma queries with multi-table JOINs that return data which rarely changes
(parent-child links, unpaid invoice list). Zero `unstable_cache` usage exists in the app today.
Same for `app/admin/page.tsx`: `totalEmployees` count re-queries on every admin dashboard visit.

Role: `cto` | Branch: `staging`

## Spec

### Acceptance Criteria

**AC-1 — Phase 3 indexes applied to production**

The 4 indexes from `prisma/migrations/20260415000000_add_perf_indexes/migration.sql` exist on
production (`qrnbanxcrmrwganpmzmn`):
```sql
CREATE INDEX "User_tenantId_status_idx" ON public."User"("tenantId", status);
CREATE INDEX "PayrollItem_payrollRunId_employeeId_idx" ON public."PayrollItem"("payrollRunId", "employeeId");
CREATE INDEX "Admission_tenantId_status_idx" ON public."Admission"("tenantId", status);
CREATE INDEX "ClassSection_tenantId_idx" ON public."ClassSection"("tenantId");
```
Verified via `SELECT indexname FROM pg_indexes WHERE schemaname='public'` on prod. Build + tests green.

**AC-2 — 8 new FK indexes in Prisma schema + migrated to both DBs**

`prisma/schema.prisma` gains (in the correct models):
```prisma
model Campus {
  @@index([tenantId])
}
model Employee {
  @@index([campusId])
}
model EmployeeSalaryValue {
  @@index([componentDefId])
}
model LeaveRequest {
  @@index([employeeId])
}
model PayrollItem {
  @@index([employeeId])   // new — separate from the composite (payrollRunId, employeeId) index
}
model PayrollItemLine {
  @@index([payrollItemId])
  @@index([componentDefId])
}
model PayrollRun {
  @@index([tenantId])
}
```
Migration generated (`npx prisma migrate dev --name add_fk_indexes`), applied to staging.
Production patched directly via Supabase MCP `apply_migration` with the same SQL.
`pg_indexes` on both DBs confirms all 8 indexes. Build + tests green.

**AC-3 — RLS initplan fixed on production**

All 15 policies that call `auth.uid()` or `current_setting()` per-row are updated to use
`(select auth.uid())` pattern. Affected policies (all `auth_rls_initplan` WARN-level):
- `Tenant.tenant_select_own`
- `User.user_select_own`, `User.user_update_own`
- `Campus.campus_select_own_tenant`
- `OrgConfig.orgconfig_select_own_tenant`
- `Holiday.holiday_select_own_tenant`
- `Employee.employee_select_own_tenant`
- `LeaveRequest.leaverequest_select_own_tenant`
- `SalaryComponentDef.salarycomponentdef_select_own_tenant`
- `EmployeeSalaryValue.employeesalaryvalue_select_own_tenant`
- `AttendanceRecord.attendancerecord_select_own_tenant`
- `PayrollRun.payrollrun_select_own_tenant`
- `PayrollItem.payrollitem_select_own_tenant`
- `PayrollItemLine.payrollitemline_select_own_tenant`
- `EmailLog.emaillog_select_own_tenant`

Applied to production via Supabase MCP. Supabase Advisor `get_advisors(type=performance)` returns
zero `auth_rls_initplan` warnings afterwards. Build + tests green.

**AC-4 — `unstable_cache` wraps parent + admin dashboard data**

`lib/parent-helpers.ts`:
```ts
import { unstable_cache } from "next/cache";

export const getParentWithChildren = unstable_cache(
  async (session: SessionUser) => { /* existing Prisma query */ },
  ["parent-children"],
  { revalidate: 300, tags: ["parent-children"] }  // 5 min TTL
);

export const getStudentInvoices = unstable_cache(
  async (studentId: string): Promise<StudentInvoices[]> => { /* existing query */ },
  ["student-invoices"],
  { revalidate: 120, tags: ["student-invoices"] }  // 2 min TTL
);
```

Cache invalidation — add `revalidateTag()` to:
- `app/api/xendit/webhook/route.ts` — on payment PAID: `revalidateTag('student-invoices')`
- Any route that creates/updates an Invoice: `revalidateTag('student-invoices')`
- Any route that updates StudentGuardian/Parent: `revalidateTag('parent-children')`

`app/admin/page.tsx`: wrap `totalEmployees` count in `unstable_cache`:
```ts
const getEmployeeCount = unstable_cache(
  async (tenantId: string) =>
    prisma.employee.count({ where: { tenantId, status: "ACTIVE" } }),
  ["employees-count"],
  { revalidate: 1800, tags: ["employees-count"] }  // 30 min TTL
);
```
Add `revalidateTag('employees-count')` to employee create/deactivate routes.

`app/admin/page.tsx` `todayAttendance` and `weeklyTrend`: NOT cached (live data, changes as
teachers check in). `pendingLeave`: NOT cached (changes on leave submissions, short loop).

Verification: repeat `curl -s -o /dev/null -w "%{time_total}" <parent-dashboard-url>` shows
the second request returns significantly faster. Build + tests green (73+ passing).

### Out of Scope

- Multiple permissive policies (`_service_all` + `_select_own_tenant` dual SELECT policies):
  consolidating these requires verifying the service role bypass chain and is a separate security
  review, not a quick fix.
- Vercel Analytics: low performance impact, separate cycle.
- Dynamic imports for remaining heavy admin pages: no recharts found; deferred since InvoiceDetailSheet
  is already done and the next-largest savings require component extraction refactors.
- `pendingLeave` + `weeklyTrend` caching: these are live-ish data; risk of stale-data bugs.
