# Roster Import — AY 2026/2027 (New Intake)

## Context

Follow-on from the pilot readiness audit ([docs/cycles/2026-07-12-pilot-readiness-audit.md](2026-07-12-pilot-readiness-audit.md)) — T1 found prod had 0 students / 0 class sections for AY 2026/2027, blocking any real pilot testing. Owner provided two roster spreadsheets:

- `Data siswa TA 2526-1.xlsx` (136 students, inconsistent per-sheet academic-year labels, no campus column) — owner decided to **drop this file entirely**, not import.
- `Data Lengkap Siswa 2026-2027.xlsx` (35 students, explicit campus/class per sheet, correct year) — the actual new-intake roster for AY 2026/2027, cross-confirmed against its own `SPMB` (prospective-student) sheet.

One student (Ghiani Harahap, `Data Lengkap TK Aster` row 10) had only name+gender filled — no class, DOB, or guardian data — excluded from this import; owner to complete her record separately.

## Spec

Import the 34 complete students into production, each enrolled in the correct campus/program/class for AY 2026/2027, with father+mother as guardians carrying full contact/bio data, such that:
- Teachers can immediately mark attendance for these students (requires `ClassSession` rows to exist for today).
- Admin can see students, guardians, and enrollments through the normal admin UI — no data invisible to existing app views.

Non-goals: parent portal accounts/login (parent portal still deferred per [[project_pilot_rollout]]), NIS/NISN assignment (not present in source file), Ghiani Harahap's record.

## Implementation

No application code changed — this cycle is pure production data setup, executed by driving the app's own existing, validated API endpoints (no bulk-import feature exists; building one was explicitly declined in favor of a one-off pass, see chat). Field mapping: Indonesian "Tempat, DD Month YYYY" birth strings parsed to `birthPlace`/`dateOfBirth`; "Kelas" column (KB / TK A / TK B) resolved to the matching `ClassSection`; address = `Alamat, Desa/Kelurahan, Kecamatan` concatenated; `Tinggal` ("Orangtua") mapped to `livingWith: ORANG_TUA`; father/mother NIK, education, occupation, employer, and income mapped onto `Parent` via the guardian-add endpoint (richer than the student-create endpoint's inline guardian shape, which only accepts name/relationship/phone/isPrimary).
