# Curriculum PROMES Import (Pack 1 / Cycle 2 of 11)

## Context

Cycle 1 ([2026-05-12-curriculum-schema-and-admin](./2026-05-12-curriculum-schema-and-admin.md), shipped via [PR #248](https://github.com/ismailir10/annisaa-erp-v3/pull/248)) delivered the curriculum data model (`Semester`, `Theme`, `SubTheme`, `Week`, `LearningObjective`, `AchievementIndicator`, `IndicatorThemeLink`) and the admin Semester / Theme / SubTheme / Week authoring UI. Schema is in place, but `LearningObjective` + `AchievementIndicator` rows are unpopulated â€” currently the only entry path is manual SQL, which won't scale to ~200 IKTPs per semester per `ageGroup`. The school authors PROMES in xlsx (`PROMES TK A SMT 1.xlsx`, `PROMES TK B SMT 1.xlsx`) and the design doc ([docs/superpowers/specs/2026-05-12-curriculum-penilaian-raport-design.md Â§6.1](../superpowers/specs/2026-05-12-curriculum-penilaian-raport-design.md)) names xlsx import as the primary seed path for TP + IKTP rows. This cycle ships that parser + admin upload surface so C3 (TP + IKTP CRUD UI, IndicatorThemeLink matrix) can land on top of populated curriculum data rather than a blank schema. Outcome: a SUPER_ADMIN uploads one PROMES xlsx per ageGroup, reviews parsed rows in a preview pane, and commits â€” `LearningObjective` + `AchievementIndicator` rows materialise atomically under the target `Semester`, audit-logged, idempotent (re-import yields a 409 with conflict report rather than silent dupe). `IndicatorThemeLink` rows are out of scope for C2 (matrix UI is C3 â€” parser detects theme columns for forward compatibility but does not write the links yet).

**Assumptions** (correct these before `/build` proceeds)

1. **Test fixtures are synthetic, committed to repo at `lib/curriculum/__fixtures__/`.** Real `PROMES TK A SMT 1.xlsx` files from school contain author cells, walas references, and live indicator text that may evolve cycle-over-cycle â€” committing them couples test stability to school workflow. Synthetic minimal fixtures (5 elements Ă— ~3 TPs each Ă— ~2 IKTPs each, ~30 total IKTPs per ageGroup) reproduce every parser edge case without PII or drift risk. Real PROMES files stay outside the repo; admin uploads them at runtime. **If you want the real PROMES files committed (e.g. for cross-cycle parity testing)**, say so now â€” that's a deliberate choice to track upstream xlsx changes in git history.

2. **`ageGroup` is an explicit form field on the upload, with filename-heuristic pre-fill.** Filename regex on `/TK\s*A/i` / `/TK\s*B/i` is fragile (PROMES authors rename copies, paste into Google Drive, version-suffix files like `PROMES TK A SMT 1 (final v2).xlsx`). The upload form has a `RadioGroup` (TK A / TK B) the admin must confirm before submit; if the filename matches the heuristic, the radio is pre-selected. Sheet-header inference also runs as a secondary fallback (parser surfaces the inferred value in the preview payload so the admin can spot a mismatch). **If you want filename-only with no UI radio**, say so â€” that removes one click but trades it for occasional silent mis-assignment.

3. **Element header tolerance is case-insensitive trimmed substring match.** PROMES authors edit copies and capitalization drifts (`NAM PROGRAM SEMESTER`, `Nam program semester`, `NAM`, `Nilai Agama dan Budi Pekerti`, `STEAM / LITERASI`). Parser normalises each header cell to uppercase + trimmed + collapsed whitespace, then runs substring match against a fixed alias table per `CurriculumElement` (e.g. `NAM` / `NILAI AGAMA` â†’ `RELIGIOUS_MORAL`). Exact match is too brittle for the artifact reality. **If you want exact-match enforcement with admin-edit-pre-commit for misspellings**, say so â€” exact match catches typos earlier but blocks every minor formatting drift.

4. **Conflict policy on re-import: hard reject 409 surfaced in preview.** No "overwrite" toggle in C2. Re-importing a PROMES that collides with existing `LearningObjective` rows under the same `(tenantId, semesterId, ageGroup, element, number)` key returns 409 with a per-row conflict list â€” admin must either pick a different `Semester` or manually clean up existing rows via C3's CRUD UI before retrying. Audit-trailed overwrite is C3's concern (matrix UI is the right place for surgical edits). **If you want an "overwrite all" force flag with explicit audit trail in C2**, say so â€” it's a one-task delta but it ships an admin-foot-gun before the CRUD UI lands to undo it.

5. **`AchievementIndicator.order` derives from xlsx row order (1-indexed, monotone within parent objective).** PROMES doesn't carry an explicit indicator-number column on IKTP rows (the number column is for TP). Row order in the source xlsx is the authoritative ordering signal. **If PROMES authors are known to reorder IKTPs without intent (e.g. alphabetised by accident)**, say so â€” we'd then need a stable id strategy independent of row order.

6. **xlsx library = `exceljs` (MIT, actively maintained on npm).** Originally specced as `xlsx` (SheetJS CE) for smaller bundle. Switched to `exceljs` mid-build after spotting that npm-registry `xlsx@0.18.5` carries CVE-2023-30533 (prototype-pollution on parse, patched only on cdn.sheetjs.com 0.19.3+). Attack surface is narrow here (SUPER_ADMIN-only upload, server-only parse, buffer never reaches the client), but defense-in-depth wins â€” `exceljs` is on the standard npm registry with a clean CVE record. Bundle size is irrelevant for a server-only parser. Merged-cell + style introspection API also maps cleanly to PROMES sheets.

7. **Audit-log shape.** One `AuditLog` row per commit, entity `curriculum.import-promes`, action `create`, before `null`, after `{ semesterId, ageGroup, objectivesCount, indicatorsCount, filename }`. Per-row audit (one row per inserted `LearningObjective` / `AchievementIndicator`) would flood the audit feed for a typical ~200-IKTP import. C3's CRUD UI emits per-row audit for surgical edits where granularity matters.

8. **Permission gate `curriculum.write` (SUPER_ADMIN only via system-role defaults), not a new `curriculum.import` code.** C1 reserved `curriculum.write` as the umbrella mutate permission; carving a separate `curriculum.import` would mean a permission-matrix doc update for one route. SUPER_ADMIN escape hatch already covers the route regardless.
