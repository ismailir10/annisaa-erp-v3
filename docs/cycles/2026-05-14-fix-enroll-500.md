# Fix Enroll POST 500 — Cutover Blocker (Finding-5)

## Context

E2E staging cycle on 2026-05-14 ([docs/cycles/2026-05-14-e2e-admission-lifecycle.md](2026-05-14-e2e-admission-lifecycle.md)) surfaced `POST /api/students/[id]/enroll` returning 500 on every valid request — confirmed on three section/age combinations, body `{"error":"Terjadi kesalahan server"}`. Blocker for July 2026 cutover because no new student can be placed in a class. Root cause: [app/api/students/[id]/enroll/route.ts:57-64](app/api/students/%5Bid%5D/enroll/route.ts:57) combines `FOR UPDATE OF cs` with `GROUP BY cs.id, cs.capacity` in the same `tx.$queryRaw`. Postgres rejects this: *"SELECT FOR UPDATE/SHARE cannot be used with aggregate functions or GROUP BY"*. The throw is caught by the route's generic `catch` and returned as 500.

Review of the first draft of this spec (feature-dev:code-reviewer agent) surfaced two additional safety gaps that share the same fix surface and should land together: (a) the route's `classSection.findUnique` at line 27 and the inner `$queryRaw` at line 57 do not filter by `tenantId`, so an admin from tenant A can pass a tenant B `classSectionId`; (b) `ClassSection.capacity` is nullable in the schema and the comparison `Number(active_count) >= section[0].capacity` returns `false` when capacity is `null`, silently bypassing the cap. The original T3 (dialog toast UX) is dropped — the `handleEnroll` at [app/admin/students/[id]/page.tsx:214-216](app/admin/students/%5Bid%5D/page.tsx:214) already renders `d.error` from the JSON body; the user's "Terjadi kesalahan server" toast in the E2E session was the literal body of the 500, not a client-side fallback.

## Spec

**Acceptance criteria:**
- [ ] `POST /api/students/[id]/enroll` returns 201 + the new `StudentEnrollment` row on a valid request (existing student, active section with capacity, age within program limits)
- [ ] Returns 400 `{error: "Kelas penuh (N/M)"}` when section is at or above capacity — exact text from existing `EnrollError`
- [ ] Returns 400 with age message when DOB outside program age range — already works, must not regress
- [ ] Returns 400 `{error: "Siswa sudah terdaftar di kelas lain..."}` when student already has an ACTIVE enrollment — already works, must not regress
- [ ] Returns 404 `"Kelas tidak ditemukan"` when the supplied `classSectionId` does not belong to the caller's tenant (tenant boundary)
- [ ] ~~Null capacity guard~~ — dropped after second read of schema: `ClassSection.capacity Int @default(20)` is **not nullable**, so the bypass reviewer flagged is impossible at the DB layer. Acceptance + test removed.
- [ ] Concurrency: two simultaneous enroll POSTs against the same section near capacity cannot both succeed. Verified by the row-lock on `ClassSection`.
- [ ] Vitest at `app/api/__tests__/enroll.test.ts` (existing file) covers: happy path, capacity full, age below min, age above max, duplicate enrollment, **cross-tenant section rejected**. Existing 3 tests updated for new mock shape (`tx.studentEnrollment.count` replaces the count returned by `$queryRaw`). 6 cases total.
- [ ] No new Prisma schema migration
- [ ] `npm run build && npx vitest run` clean

**Non-goals:**
- No change to admission, attendance, invoice, or graduate routes
- No refactor of other `$queryRaw FOR UPDATE` callsites in the repo (audit deferred — flag any matches as a follow-up cycle)
- No change to `@@unique([studentId, classSectionId])` or other schema constraints
- **Soft-deleted / non-ACTIVE student enrollment guard** — reviewer flagged that a `WITHDRAWN` or `GRADUATED` student can currently be re-enrolled. Deferred: design decision (re-enrollment may be a legitimate flow). Separate cycle if Cycle 2 confirms it should be blocked.
- **INACTIVE class section enrollment guard** — same shape; deferred to a separate cycle.
- No fix for the other 7 findings (F-1, F-2, F-3, F-4, F-6, F-7, F-8) — separate cycles

**Assumptions:**
1. Supabase pooler in transaction mode does support `FOR UPDATE` within a single `prisma.$transaction` callback — the bug is the SQL combination, not the pooler. Confirmed: PgBouncer transaction mode keeps a single connection for the transaction's lifetime; row locks inside that transaction work.
2. The existing `EnrollError` class + status-aware catch is correct; only the SQL inside the try-block changes.
3. `StudentEnrollment.@@unique([studentId, classSectionId])` exists (verified in schema:553) and provides the duplicate-row guard at DB level even if the application check is bypassed.
4. Concurrency guarantee, scoped: splitting the lock query (`SELECT cs FOR UPDATE`) from the count query (`tx.studentEnrollment.count`) inside one `$transaction` preserves the lock for the count's duration — Postgres holds row-level locks until tx commit/rollback. **This blocks other transactions that also lock the same `ClassSection` row first.** Every existing enrollment path in the repo goes through this route, so all concurrent callers serialize. The guarantee is *not* absolute: a future code path that inserts a `StudentEnrollment` row without first acquiring the `ClassSection` row lock would slip past. If a bulk-import or admin-override route is added later, it must take the same lock; otherwise consider a Postgres advisory lock keyed by `classSectionId` or a `CHECK` constraint counter.

- **T1** — Edited [app/api/students/[id]/enroll/route.ts](app/api/students/%5Bid%5D/enroll/route.ts) only. Diff ≈22 lines. Locking semantics preserved (row lock held until tx commit). Added a 6-line comment explaining the SQL restriction + lock contract for future readers.
