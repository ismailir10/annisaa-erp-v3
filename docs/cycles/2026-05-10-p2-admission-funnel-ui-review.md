# P2 Admission Funnel UI — Admin Review Half (transitions + admin screen + accept side-effect bundle + emails)

## Context

`p2-admission-funnel-ui-public` shipped (PR [#214](https://github.com/ismailir10/school-erp/pull/214) `1e7f945`, §18A flipped via [#215](https://github.com/ismailir10/school-erp/pull/215) `3db0d1c`). Two follow-up hotfixes landed on staging — [#216](https://github.com/ismailir10/school-erp/pull/216) (OAuth callback preview-fallback + login embedded on `/`) and [#217](https://github.com/ismailir10/school-erp/pull/217) (`/admin` landing card grid). Staging tip = `235a79b`. Live smoke confirmed: real admin OAuth flow signs `ismailir10@gmail.com` in via Google, lands on `/admin`; `/daftar` POSTs an `Admission` row in `SUBMITTED`.

This cycle delivers the deferred half of the original `p2-admission-funnel-ui` scope — every admin-facing surface that consumes the SUBMITTED row and walks it through the rest of the 8-state machine. Foundation §18.1 row order positions `p2-admission-funnel-ui-review` as the next sequential cycle.

**§18.2 cap pre-emptively split.** Full admin-half scope (4 transition actions + ACCEPTED side-effect bundle + admin review screen + 3 detail tabs + MPLS list + MPLS cohort detail + bulk save-attendance action + 2 email templates + admin Playwright spec + entity registry + permission seed for MPLS) estimated at ~28 files. Same cap-split rationale as the prior cycle. Two natural cuts:

- **This cycle (`p2-admission-funnel-ui-review`)** — admin transitions (review/accept/reject/withdraw) + admin review screen with state-aware actions + 3 detail tabs (ringkasan/assessment/aktivitas) + ACCEPTED side-effect bundle (Household + Student + Guardian + StudentGuardian creation in single tx) + admission-accepted/rejected emails + admin Playwright spec. Closes the admission-funnel write-loop end to end. Estimated ~17 files staged.
- **Follow-up (`p2-mpls-minimal-admin`)** — MPLS cohort entity registry + permission seed + cohort list page + cohort detail page with attendance grid + `saveAttendance` bulk action + vitest. Independent of the admission funnel — MPLS happens AFTER admission ACCEPTED, no FK or transition coupling. Ships as its own ~10-file cycle.

**Why split here.** MPLS UI is orthogonal to the admission funnel — it depends on `MplsMember` rows which only exist after admin manually enrolls accepted students into a cohort (a separate admin workflow). Bundling MPLS into this cycle would mix two reviewable scopes into one PR + risk landing the admission close-loop behind unrelated MPLS plumbing. Splitting also keeps each PR under the 25-file soft-cap.

## Spec

### Acceptance criteria

- [ ] **AC1 — Four transition action files.** Create `lib/admission/transitions/{review,accept,reject,withdraw}.ts`. Each mirrors `submit.ts` shape: wraps `assertTransition` from `lib/admission/state-machine.ts` + `assertScope` per `lib/scaffold/server-action.ts` + `writeAuditLog` (action=UPDATE, before/after status snapshots) + `emitTimelineEvent("admission.status-changed", payload={from, to, reason?})`. Session resolved via `getSession()` (admin / principal / kadiv / admission_officer). Audit + timeline writes inside `prisma.$transaction`. Per-action vitest covers happy path + illegal-transition rejection + scope rejection + audit row + timeline event.
- [ ] **AC2 — `accept.ts` ACCEPTED side-effect bundle.** Single `prisma.$transaction` runs:
  1. Resolve target Household: if `Admission.siblingDetectedFromHouseholdId` is non-null, **load that Household + its existing Guardians**; else create a new Household scoped to the resolved tenant + linked to `Admission.addressId`.
  2. Create Student row with applicant snapshot fields (`fullName`, `nickname`, `nik`, `birthDate`, `gender`, `birthPlace`) → set `Admission.acceptedStudentId`.
  3. **NIK-merge Guardians.** For each non-empty parent snapshot (father, mother): if a Guardian with matching NIK already exists in the resolved Household, reuse that Guardian (skip create); else create a new Guardian row.
  4. Create StudentGuardian links (FATHER + MOTHER, both `isPrimary=true` per the relationship-scoped partial-unique guard from #192).
  5. Transition Admission to ACCEPTED + audit row (UPDATE, before={status:OFFER_EXTENDED}, after={status:ACCEPTED, acceptedStudentId}) + timeline event.
  Vitest covers: new-Household path (no sibling FK), sibling-merge path (skip Household create + reuse existing matching Guardian), transactional rollback on any sub-write failure (e.g. Guardian P2002 unique violation outside the merge path), idempotent against double-click (re-running on already-ACCEPTED row throws `INVALID_TRANSITION` not duplicate side-effect).
- [ ] **AC3 — Admin review screen.** `app/admin/akademik/penerimaan/[id]/page.tsx` (server) + `client.tsx` (client). Server fetches the Admission row + denormalized parent snapshots + sibling Household label (when FK populated) + acceptedStudent row (when FK populated) + InitialAssessment rows + recent TimelineEvent rows (subjectKind=Admission). Client renders state-aware action buttons per the 15-edge graph:
  - SUBMITTED → "Pindahkan ke review" (UNDER_REVIEW) / "Tolak" / "Tarik kembali"
  - UNDER_REVIEW → "Jadwalkan wawancara" (INTERVIEW_SCHEDULED, opens dialog with `<Input type="date">`) / "Tawarkan tempat" (OFFER_EXTENDED, fast-track) / "Tolak" / "Tarik kembali"
  - INTERVIEW_SCHEDULED → "Tawarkan tempat" / "Tolak" / "Tarik kembali"
  - OFFER_EXTENDED → "Tandai diterima" (ACCEPTED — confirmation dialog "Akan membuat 1 Keluarga + 1 Siswa + N Wali baru. Lanjutkan?") / "Tolak" / "Tarik kembali"
  - ACCEPTED / REJECTED / WITHDRAWN → no action buttons (terminal). Layout follows `.claude/standards/patterns.md` Recipe 2 (PageHeader + breadcrumbs + Tabs + right-rail metadata aside). Three deferred detail tabs from #211 wired: `ringkasan` (applicant + parent + address read-only), `assessment` (read-only InitialAssessment list), `aktivitas` (TimelineEvent feed filtered subjectKind=Admission AND subjectId=`<id>`). Sibling-detected Household surfaces as `Detected sibling: <household_label>` badge in the page header. Indonesian copy in Bu Sari voice per `.claude/standards/voice.md`.
- [ ] **AC4 — Email templates `admission-accepted` + `admission-rejected`.** Two new files under `lib/email/templates/` registered in the typed-slug barrel (`lib/email/templates/index.ts`). `admission-accepted.tsx` renders Indonesian Bu Nur voice — Assalamu'alaikum opener + welcome + next-steps stub (placeholder for MPLS schedule + Akta Kelahiran upload reminders) + tracking code + new student name. `admission-rejected.tsx` renders polite Indonesian (no specific reason — admin can edit `Admission.notes` later for free-text). ### Assumptions (correct now or `/build` proceeds)

1. **MPLS pre-split.** This cycle ships `-review` only. MPLS lands as `p2-mpls-minimal-admin` in the next sequential cycle slot. If user wants MPLS bundled here, override now and accept the >25-file PR (~28 files). The §18A row prepended this cycle is `p2-admission-funnel-ui-review` only.
2. **Sibling-merged Household: NIK-merge Guardians, not duplicate.** When `Admission.siblingDetectedFromHouseholdId` is populated and the existing Household already has a Guardian with matching NIK (e.g. mother's NIK matches an existing mother Guardian on a previously-accepted sibling), the ACCEPTED bundle **reuses** the existing Guardian (skip create) and only creates a new StudentGuardian link. If NIK is empty on the snapshot (parent skipped NIK on the public form), a new Guardian row is created. No cross-Household NIK guard — Guardian NIK is unique per (tenantId, householdId) per the schema; cross-Household NIK collisions are out of scope.
3. **INTERVIEW_SCHEDULED date dialog uses `<Input type="date">`.** Native browser date picker — no Calendar component reuse this cycle.
